## 1. Goal

Add `update-contact` to `contact-mutations.ts` and `ContactManagerPanel`. A MANAGER edits an existing contact's fields; the preferred-method pointer is preserved unless the edit clears that method's field while another method remains populated, in which case the edit is rejected outright — reassignment happens only through the explicit `set-preferred-contact-method` action, not as a side effect of a field edit. Visible result: editing a contact's details succeeds and is audited without silently disturbing which method is preferred; an edit that would strand the preferred-method invariant is blocked with a clear message.

## 2. Design

### 2.1 Preferred-method resolution on update — three cases, not a simple carry-over

`cm11`'s `resolvePreferredMethod` only ever runs against a **brand-new** contact (nothing to preserve). Update needs a genuinely different, three-way resolution against the contact's **existing** `preferredContactMethod`:

```ts
function resolveUpdatedPreferredMethod(
  current: PreferredContactMethod | null,
  updated: ContactFields,
): { ok: true; value: PreferredContactMethod | null } | { ok: false } {
  const populated = {
    PHONE: updated.phoneNumber !== null,
    EMAIL: updated.emailAddress !== null,
    ADDRESS: updated.addressLine1 !== null,
  } as const

  if (current === null) {
    // Nothing was preferred before — same first-populated priority as cm11's create path.
    if (populated.PHONE) return { ok: true, value: 'PHONE' }
    if (populated.EMAIL) return { ok: true, value: 'EMAIL' }
    if (populated.ADDRESS) return { ok: true, value: 'ADDRESS' }
    return { ok: true, value: null }
  }

  if (populated[current]) return { ok: true, value: current } // still populated — unchanged, no reassignment
  const anyOtherPopulated = (Object.keys(populated) as PreferredContactMethod[])
    .some((k) => k !== current && populated[k])
  if (anyOtherPopulated) return { ok: false } // Module Inv. #5 — blocked, not auto-reassigned
  return { ok: true, value: null } // clearing down to zero populated methods — fine
}
```

- **Nothing preferred yet, and this edit populates a method for the first time** ⇒ same priority as creation (phone > email > address).
- **The currently-preferred method's field is still populated after the edit** ⇒ untouched, no reassignment logic runs at all (this is the common case — most edits don't touch the preferred field).
- **The currently-preferred method's field is being cleared, and another method remains populated** ⇒ **blocked**. This is Module Inv. #5's literal wording ("clearing the preferred method is blocked while another method is still populated") — the fix is to call `set-preferred-contact-method` (not built in this pass) first, then retry the edit that clears the old one.
- **The currently-preferred method's field is being cleared, and nothing else is populated either** ⇒ allowed, `preferredContactMethod` becomes `null` (Inv. #5's "NULL iff no method is populated" is satisfied, not violated).
- **No separate `PREFERRED_METHOD_CHANGED` audit event for the "nothing preferred yet" auto-assignment case** — same economy `cm11` used for creation; `CONTACT_UPDATED`'s `afterData` already shows the new value. The dedicated event type is reserved for the explicit action.

### 3.3 Service — `contact-mutations.ts` (extend, add `updateContact`)

```ts
export async function updateContact(input: UpdateContactInput, actorId: string): Promise<
  | { ok: true; value: { lastModifiedDatetime: Date } }
  | { ok: false; code: 'CONFLICT' | 'CONTACT_NOT_FOUND' | 'PREFERRED_METHOD_STILL_POPULATED' }
> {
  const before = await contactMediumRepository.findById(db, input.contactMediumId)
  if (before === null || before.refPartyRole !== input.partyRoleId) return { ok: false, code: 'CONTACT_NOT_FOUND' }

  const resolved = resolveUpdatedPreferredMethod(before.preferredContactMethod, input)
  if (!resolved.ok) return { ok: false, code: 'PREFERRED_METHOD_STILL_POPULATED' }

  return db.transaction(async (tx) => {
    const bumped = await partyRoleRepository.compareAndBumpLock(tx, input.partyRoleId, input.lastModifiedDatetime)
    if (bumped === null) return { ok: false, code: 'CONFLICT' }

    const after = await contactMediumRepository.update(tx, input.contactMediumId, {
      ...input, preferredContactMethod: resolved.value, lastModifiedBy: actorId,
    })

    await writeAuditEvent(tx, {
      eventType: 'CONTACT_UPDATED', actorUserId: actorId, targetEntity: 'CONTACT_MEDIUM',
      targetId: input.contactMediumId, beforeData: before, afterData: after,
    })

    return { ok: true, value: { lastModifiedDatetime: bumped } }
  })
}
```
