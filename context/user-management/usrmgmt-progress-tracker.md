### Deviations from the spec's literal text

- **Drizzle PK properties renamed `userId`/`accountId`/`sessionId`/`verificationId` → `id`.** The installed Better-Auth adapter (`1.6.19`) hard-codes the lookup of a literal `id` field for every model (`getDefaultFieldName`: `if (field === "id") return "id"`, never remappable via `fields`) and the Drizzle adapter then expects `schemaModel.id` to exist as a JS property — there is no `fields.id` escape hatch (the type explicitly excludes `"id"` from the map). Renamed the four PK properties in `db/schema/identity.ts` to `id`; the underlying SQL column names (`user_id`, `account_id`, `session_id`, `verification_id`) are unchanged, so no new migration was needed and `tests/db/identity-schema.test.ts` (which asserts on `.name`, not the property key) needed no change. This _is_ the field mapping spec §3.7 describes (`id → user_id`); the installed version just requires it expressed as a property rename rather than a `fields` config entry.
- **`audit_log.actor_user_id` is `text`, not `uuid` (spec §3.2's literal table).** The FK target, `core.appuser.user_id`, is `text` (Better-Auth's own id format, established in um02) — a `uuid`-typed FK column referencing a `text` PK is not valid DDL. `text` matches the column it actually references; `target_id` was already `text` for the same reason.
- **Field mapping only remaps fields whose Better-Auth default name differs from our Drizzle property** (e.g. `user.name → userName`, `account.accountId → providerAccountId`, every model's `createdAt`/`updatedAt` → `createdDatetime`/`lastModifiedDatetime`). Fields already named identically (`session.userId`, `session.expiresAt`, `account.providerId`, `verification.identifier`, …) need no entry — Better-Auth's `fields` option maps its field name to _our Drizzle schema's property key_ (which already encodes the snake_case SQL column via `text("user_name")` etc.), not to a raw column string, so listing them would be a no-op.
- **Status check lives in `databaseHooks.session.create.before`, not a `checkCredentials`/`authorize` callback.** The installed `emailAndPassword` option type has no such callback; `signInEmail`'s own route source verifies the password and then calls `internalAdapter.createSession`, which is the only model-create operation gated by `databaseHooks` before the row commits. - **Audit/last-login hook lives in `databaseHooks.session.create.after`, not an `emailAndPassword.onSignIn` callback** (the installed version has no such callback). Confirmed via `with-hooks.mjs`/`transaction.mjs` that `queueAfterTransactionHook` is awaited before `internalAdapter.createSession` (and thus `auth.api.signInEmail()`) returns, so the hook has always run by the time a caller observes the result — verified directly against a live Postgres container (session, audit row, and `last_login_datetime` all present immediately after `await auth.api.signInEmail(...)`).
- **No shadcn `Form` component in the installed CLI version (`shadcn@4.11`, "Nova" preset).** `npx shadcn add form` is a silent no-op in this registry; the replacement composition is `Field`/`FieldLabel`/`FieldContent`/`FieldError`/`FieldGroup` (added via `npx shadcn add field`, which also pulled in `separator.tsx`), not bound to React Hook Form via context — `LoginForm` wires it with plain `register()`/`formState.errors` instead of `FormField`/`Controller`.
- **`db/client.ts`'s `Database` type widened to a union** (`typeof db | PostgresJsTransaction<...>`). A `db.transaction(async (tx) => …)` callback's `tx` lacks the pool-only `$client` property, so under `exactOptionalPropertyTypes` it didn't structurally satisfy the um02 `Database = typeof db` alias — needed for `auth/index.ts`'s hook to pass `tx` into `updateLastLogin`/`insertAuditEvent` (exactly the "accepts either the pool `db` or a transaction `tx`" contract um03 §3.5/§3.6 specify).
- **`eslint.config.mjs` boundary additions required by the spec's own file layout.** `components/login-form.tsx` must import `authClient` from `auth/client.ts` (um03 §3.8) and `loginSchema` from `validation/` (code-standards §4.12), neither allowed by um01/um02's matrix. Carved out a separate `auth-client` element (pattern `auth/client.ts`, ordered before the general `auth/**` pattern) so `components/**` can reach only the client-safe entry point — never `auth/index.ts` — and added `validation` to `components`'s allow-list.

## um04 — Custom Per-Account Lockout (LOCAL)

**Spec:** `context/user-management/specs/um04-lockout.md`

### Scope

Per-account lockout for LOCAL sign-in only: a repository-backed state machine wired into the existing Better-Auth sign-in flow. Reject already-locked accounts before password verification, increment `failed_login_count` on each wrong password, lock for 15 minutes + `USER_LOCKED` audit on the 5th consecutive failure, reset on success. ### Deviations from the spec's literal text

This unit required substantially more spec-vs-installed-version reconciliation than any prior unit. Read `node_modules/@better-auth/core/src/oauth2/link-account.ts` and `node_modules/better-auth/dist/api/routes/{sign-in,callback}.mjs` directly (not just the type definitions) to confirm the following before implementing:

- **No `/administration/system-config` page existed before this unit.** um10-spec §10.8 assumes one ("the existing… page gains a section") but no unit through um09 built it — `components/admin-nav.tsx`'s link to it was a deliberate 404-until-shipped placeholder since um07. A future unit that adds real system-config content should treat this page as already-owned, not re-create it.
- **Better-Auth's native account-linking flow already resolves the email match — our hook validates, it doesn't resolve.** The spec's design (§10.3) assumes `databaseHooks.account.create.before` looks up the Entra email itself (`findSsoUserByEmail`) and returns `{ data: { userId: existingUser.userId } }`. Reading `link-account.ts`'s `handleOAuthUserInfo` directly: Better-Auth's own `findOAuthUser(email, accountId, providerId)` already matches the email (case-insensitively, against whatever the adapter's `user.email` column is) and calls `internalAdapter.linkAccount({ userId: dbUser.user.id, ... })` _before_ `account.create.before` ever fires — there is no email available at that hook point, only the already-resolved `userId`. Consequently:
  - **`findSsoUserByEmail` was not implemented** — it would be dead code; nothing in the real flow has an email to call it with. The equivalent exclusivity check (`auth_method = 'SSO'` and not `DELETED`) is done via the existing `findUserById(db, account.userId)` instead, inside `rejectNonSsoAccountLink`.
  - **`activateSsoUser`/`updateLastLoginDatetime` were not added** — they would be identical in behavior to the existing um09 `activateUser`/`updateLastLogin`, which `handleSsoSignIn` reuses directly.
  - **No JIT provisioning is enforced via `socialProviders.microsoft.disableSignUp: true`**, not by the `account.create.before` hook. Reading `link-account.ts` further: when `findOAuthUser` finds _no_ user at all (the "totally unmatched email" case), Better-Auth's flow is `createOAuthUser`, which inserts a **new `appuser` row first**, then the `account` row — meaning a hook at `account.create.before` fires _too late_ to prevent JIT creation for that specific case (a brand-new `appuser` row would already exist). `disableSignUp: true` on the provider config makes `handleOAuthUserInfo` return `{ error: "signup disabled" }` immediately instead, before any row is touched. `rejectNonSsoAccountLink`'s job is therefore narrower than the spec's: reject a match against a LOCAL or DELETED user, not resolve or reject on email at all.
  - **`account.accountLinking.trustedProviders: ["microsoft"]` was required** for legitimate SSO sign-ins to link at all. Microsoft Entra ID does not return the `email_verified` claim unless separately configured as an optional claim in the app registration (confirmed in the installed provider's own source comments), so without `trustedProviders`, Better-Auth's default implicit-linking gate (`!isTrustedProvider && !userInfo.emailVerified`) would reject every real Entra sign-in as "account not linked" — not in the spec's implementation steps, but necessary for the feature to work at all under this Better-Auth version's default account-linking policy.
- **Three distinct rejection query-string codes are possible, not one literal `sso_no_account`.** Given the above, "no email match at all" surfaces as `error=signup_disabled` (Better-Auth's own `disableSignUp` rejection), our exclusivity hook's throw surfaces as `error=unable_to_link_account` (the generic message `handleOAuthUserInfo`'s catch block substitutes for _any_ thrown error from `linkAccount`, so a custom thrown code/message does not survive to the redirect), and Better-Auth's own account-linking-policy block (shouldn't normally trigger given `trustedProviders`, but is a real reachable path) surfaces as `error=account_not_linked`. The login page maps all of `sso_no_account` (kept for forward-compatibility), `signup_disabled`, `unable_to_link_account`, and `account_not_linked` to the identical "not authorized" message — satisfying the spec's actual non-enumeration intent even though the literal query value differs from its one example.
- **`mapProfileToUser: (profile) => ({ id: profile.oid })`** added to the Microsoft provider config. The installed provider's default `getUserInfo` sets the user id from the `sub` claim, not `oid`; the spec explicitly prefers `oid` ("stable across token issuances") and `mapProfileToUser`'s return value is spread last over the default shape, making this a one-line override rather than a custom `getUserInfo`.
- **No GET `/api/auth/signin/microsoft` route is auto-registered by the installed Better-Auth version** (confirmed by reading `sign-in.mjs`) — only `POST /sign-in/social` (JSON body, consumed by the client SDK via `window.location`, not an HTTP redirect). The spec's literal href appears to assume a NextAuth.js-style convention that doesn't exist here. Added a dedicated Route Handler at that exact path (§10.7's "implementer note" anticipated needing to verify Better-Auth's actual routing) that calls `auth.api.signInSocial(..., asResponse: true)` and converts the result into a real 307 + forwarded `set-cookie`, preserving the spec's explicit "plain `<a>` tag, full top-level redirect, not a client-side nav" requirement. `eslint`'s `@next/next/no-html-link-for-pages` was suppressed on that one line with an inline justification — switching to `next/link`'s `<Link>` was considered and rejected: a target outside the app's page manifest risks the client router treating it as an RSC fetch instead of a real navigation.
- **Test paths follow the codebase's established `tests/<area>/` + `*.integration.test.ts` convention**, not the spec's literal `tests/unit/`/`tests/integration/` paths — same documented deviation as every unit since um04. `handleSsoSignIn`'s unit tests were added to the existing `tests/services/users-auth.service.test.ts` (alongside `setPassword`) rather than a new file, matching that file's established per-service-module convention.

### Deviations from the spec's literal text

- **Validation schema filenames are `validation/disable-user.schema.ts`/`validation/enable-user.schema.ts`, not the spec's literal `validation/users.ts`.** Same code-standards §"File naming" rule (`<entity>-<operation>.schema.ts`) already applied as a deviation in every unit since um08; followed the established convention.
- **`deleteUserSessions` was not added — the existing um06 `sessionRepository.deleteByUserId(db, userId): Promise<void>` is reused instead**, not a new same-purpose function returning a deleted-row count as the spec's §13.2.2 literally specifies. The count is explicitly "not exposed in the action result" per the spec's own text, and nothing in `disableUser` branches on it — `deleteByUserId` already does the exact `DELETE … WHERE user_id = $1` this unit needs and already accepts a transaction handle. Same "verify it already exists before adding a duplicate" precedent as um11's `findUserById` and um12's `findAllRoles`.
- **`countRemainingAdmins`/`userHasAdminRole` take `db: Database` as an explicit first parameter**, not the spec's literal no-`db`-param signatures that read the module-level `db` client directly. Every repository function in this codebase takes the handle as an explicit first argument (um06 precedent, reaffirmed in um11/um12) so callers can compose reads/writes against the same client the caller holds; repository files never import `db/client` directly.
- **`countRemainingAdmins` excludes `DISABLED` admins, deliberately diverging from um12's `roleAssignRepository.countNonDeletedUsersWithRole`** (which treats `DISABLED` as still ADMIN-_capable_ for the revoke-role guard, since a disabled admin can be re-enabled). um13's guard answers a different question — "if I disable this admin, does a _sign-in-capable_ admin remain right now" — so a `DISABLED` admin, who cannot sign in, does not count as remaining. This matches the spec's own verification checklist line ("DISABLED and DELETED ADMIN users are excluded from the remaining-admin count") precisely; it does not match um12's same-shaped-sounding function, which answers a structurally different question. Documented here since the near-duplicate function names invite confusing the two.
- **A new `actionError` state, distinct from um11's existing `localError`.** The spec's design implies one shared error state covering edit-mode's `USER_NOT_FOUND` and disable/enable's three codes, but `UserDetail` already had a tested `localError: 'USER_NOT_FOUND' | null` scoped to edit-mode from um11. Reusing it would have required widening its type and updating every existing edit-mode test's assumptions for no behavioral benefit (the two error surfaces are gated on mutually exclusive `mode` values and never render simultaneously). Added a sibling `actionError` state instead — zero risk to um11's already-passing tests.
- **The view-mode inline alert for `actionError` is additionally gated on `!isConfirmOpen`**, not spec'd explicitly. Without this, a `LAST_ADMIN` response (which keeps `isConfirmOpen` true and `actionError` set) rendered the same message in the panel _and_ inside the still-open dialog simultaneously — confirmed by a failing component test (`"Found multiple elements with the text"`). Gating the panel-level alert on the dialog being closed reproduces the spec's actual intent (`LAST_ADMIN` is read _inside_ the dialog; the panel-level alert is for the dialog-closed error codes only) without changing any other observable behavior.
- **The "Disable user" confirm button is a plain `Button`, not the vendor `AlertDialogAction`**, matching the spec's own literal JSX (§13.5) rather than typical shadcn `AlertDialog` usage. Radix's `AlertDialogAction` closes the dialog unconditionally on click; this unit's requirement that the dialog _stay open_ on `LAST_ADMIN` (so the admin can read the explanation) is incompatible with that built-in auto-close, so the spec's own markup already avoids it.
- **Button border colors use `--color-danger-500`/`--color-success-500`, not the spec's literal `--color-danger-300`/`--color-success-300`.** This design system's danger/success scales (`app/globals.css`) only define `-50`/`-500`/`-700` steps — there is no `-300` step to reference. Used the closest existing step (`-500`) rather than inventing a new token; text/hover-background still use the spec's intended `-700`/`-50` pair.
- **Test paths follow the codebase's established `tests/<area>/` + `*.integration.test.ts` convention** (`tests/validation/`, `tests/services/`, `tests/actions/`, `tests/components/`, `tests/db/`), not the spec's literal `tests/unit/`/`tests/integration/` paths — same documented deviation as every unit since um04. The repository-level coverage lives in the existing `tests/db/appuser-repository.integration.test.ts` (live-DB), not a new mock-based `tests/unit/db/app-user-repository.test.ts` — no prior unit introduced a mock-based repository test file, and this codebase's established pattern verifies repository query shape against a real Postgres instance instead.
